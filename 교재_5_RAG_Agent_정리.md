# 1️⃣ RAG 개념과 아키텍처 (p.1 ~ p.40)

## 1.1 LLM 생성 결과물의 한계점 (p.2 ~ p.9)

### LLM의 기능 (p.2 ~ p.5)

- **확률 기반 생성 모델**: 다양한 데이터로 학습한 Token으로 질의응답 수행
- 패턴뿐 아니라 **지식의 수준도 높음**
- 예시: User가 "'인셉션'의 감독이 누구야?"라고 질문하면 → LLM이 **지식에 기반해** "크리스토퍼 놀란입니다." 라고 답변
- 그러나 SLM이나 기타 모델의 경우 **정확하지 않은 답변 패턴** 발생 가능
- 패턴은 학습했지만 정확도가 떨어지는 경우: 대답은 자연스럽지만 **결과가 틀리거나 정확도가 떨어지는 문제 현상**

### 답변의 오류: Hallucination (p.6 ~ p.8)

- **Knowledge cutoff**: 해당 모델이 공부한 최신 데이터의 시점
  - 예) GPT: 2024.06, Claude: 2025.01
  - → 이 기간의 정보를 갖고 있음
- **최신 데이터, 도메인 특화 데이터, 내부 데이터** 등에서는 좋은 성능을 내기 어려움
- 예시: "올해 나온 최신 LLM 모델 알려줘" → LLM이 "GPT-4o가 최신 모델입니다!" (2024년 6월 기준의 최신 정보)
- 해결 방안: **Tool Calling**을 통해 Current data load → Search → Result를 수행하여 최신 정보 제공

### 이번 차시의 학습 내용 (p.9)

- 효과적인 Data의 입력
- 필요한 Data의 선택과 추출
- 기존 LLM의 정확한 답변 생성 능력 향상

---

## 1.2 검색 증강 생성: 검색 + In-Context Learning (p.10 ~ p.12)

### RAG의 정의 (p.10)

- **RAG = Retrieval(검색) + Augmented(증강) + Generation(생성)**
- LLM의 **In-Context Learning**: 입력 데이터를 이해하고 활용하는 능력

### Hallucination 문제와 RAG (p.11)

- 예시: "올해 나온 최신 LLM 모델 알려줘" → LLM이 "GPT-4o가 최신 모델입니다!" (2024년 6월 기준 최신 정보)

### Retrieval Augmented Generation 구조 (p.12)

- 사용자 **질문** → LLM이 **외부 DB/웹, 내부 DB, SQL DB** 등에서 **관련 정보 검색 후 전달** → **Prompt의 증강** → 답변 생성

---

## 1.3 RAG 예시 프롬프트 (p.13 ~ p.17)

### Instruction + Context + Question 구조 (p.13)

프롬프트 예시:

```
다음의 [Context]를 참고해 [Question]에 답하세요.
[Context]의 정보만으로 답변할 수 없는 경우, '답변할 수 없습니다'를 출력하세요.
---
[Context]:
김지찬 / 개요 삼성 라이온즈 소속 우투좌타 중견수. 주 포지션은 2루수 및 중견수이다.
전형적인 슬랩 히터이며 컨택, 선구안에 강점을 보인다. 통산 도루 성공률 1위(89.1%)를
기록할 정도로 발이 빠르다
2024년 외야수로 전향하며 팀의 주전 중견수로 정착했다.
---
[Question]:
김지찬의 통산 도루 성공률이 얼마나 돼?
```

### 지시사항의 중요성 (p.14)

- "Context에는 없지만..." → **RAG에서는 내부 정보가 중요하므로 해당 시나리오는 좋지 않음**

### 이번 차시의 학습 내용 (p.15)

- 답변할 수 없는 정보에 대한 지시사항을 Prompt로 설정해야 함
- 할루시네이션을 완화하고, 최신 데이터/내부 데이터를 활용하며 답변
- 답이 없는 경우 데이터에 해당 정보가 없는 것을 확인하기 위한 목적
- 데이터가 있는 경우에도 답을 하지 못하는 경우, 해당 데이터의 입력 방식이 잘못됨을 확인하기 위함

### Context와 Question의 하이라이트 (p.16 ~ p.17)

- Context 내 **통산 도루 성공률 1위(89.1%)** 부분과 Question의 **"김지찬의 통산 도루 성공률이 얼마나 돼?"** 부분이 매칭됨
- **RAG: 최신 Data와 내부 Data를 활용해 답변하는 과정**

## 1.4 RAG VS 파인 튜닝 (p.18 ~ p.27)

### RAG와 파인 튜닝 비교 (p.18)

| 구분 | RAG | 파인 튜닝 |
|:---:|:---:|:---:|
| 방식 | In-Context Learning | Learning |
| 특징 | Prompt에 해당 정보가 있을 때만 적용되므로, **학습에는 해당되지 않음** | 모델 파라미터를 직접 변경 |

### 지식 저장과 입력 방식 (p.19)

| RAG | 파인 튜닝 |
|:---:|:---:|
| 외부 DB 활용 | 모델 파라미터 변경 |
| → 다른 결과 출력 | → 다른 결과 출력 |

### 작업 방식의 차이 (p.20)

| RAG | 파인 튜닝 |
|:---:|:---:|
| **벡터 DB 구축** | **GPU resource + 학습 시간** |
| 상대적으로 간단 | **보다 많은 시간과 작업 비용 소요** |

### 데이터의 변경과 확장 방식 (p.21 ~ p.26)

**RAG의 장점:**
- DB의 내용이 바뀌어도 **LLM은 적응 가능** (p.21)
- **실제 문서가 들어가므로 환각 감소** (p.24)

**파인 튜닝의 단점:**
- Data 변경 시 **Machine-unlearning이 어려움** (p.22)
- 비식별화 처리 → **확장의 어려움** (p.23)
- 학습이 잘 되어도 **환각이 생길 수 있음** (p.25)

**RAG의 단점:**
- Search와 Context 처리에 **긴 시간 소요** (p.26)

### RAG와 파인 튜닝의 결합 (p.27)

- 파인 튜닝: **Domain 지식 학습** + RAG: **정보 보완**
- **RAG의 핵심: 정확한 Context 입력**

## 1.6 Context를 어떻게 구성하는가? (p.30 ~ p.34)

### 임베딩 벡터/키워드 기반의 유사도 Top K Search (p.30)

- 임베딩/인덱싱을 통해 텍스트를 벡터화
- 벡터 사이의 비교를 통해 선택

### 임베딩 벡터의 개념 (p.32)

- Text → **임베딩 알고리즘** → 벡터 → **임베딩 벡터**
- 임베딩 간의 비교를 통해 **벡터 사이의 관계 확인**

### 인덱싱의 개념 (p.33)

- Input 데이터의 Keyword ↔ Output 데이터의 Keyword 매칭

### 임베딩과 인덱싱 (p.34)

| 임베딩 | 인덱싱 |
|:---:|:---:|
| **의미 기반 검색** | **키워드 기반 검색** |

## 1.7 RAG의 5-step Process (p.35 ~ p.40)

### RAG 단계별 개념과 정의 (p.35)

| 단계 | 설명 |
|:---:|------|
| **Indexing** | DB 구성 |
| **Processing** | 질의 전처리 |
| **Searching** | 질의와 적합한 데이터 검색 |
| **Augmenting** | RAG 프롬프트 생성 |
| **Generating** | LLM 답변 생성 |

### RAG의 기본 파이프라인 (p.36 ~ p.39)

- 5단계가 **RAG의 기본 파이프라인**
- 모든 데이터가 인간과 소통할 수 있음 → **LLM이 가진 훌륭한 언어 능력** 활용
- **전체 단계에서 LLM이 역할 수행**
- Processing ~ Augmenting 단계에서의 개선 → **Advanced RAG**

### RAG의 기본 Pipeline 아키텍처 (p.40)

**Preprocessing 단계:**
- Corpus → Chunks → Embedding model → Embeddings → Vector database
- Corpus → Chunks → TF-IDF → TF-IDF encodings → TF-IDF index

**Runtime 단계:**
- Query → Vector database / TF-IDF index → **Rank fusion** → Top K chunks → **Generative model** → Response

# 2️⃣ RAG 데이터 전처리 (1) (p.41 ~ p.60)

## 2.1 Indexing: 데이터 준비하기 (p.42 ~ p.51)

### Indexing 개요 (p.42)

- 데이터를 준비하는 과정
- 데이터를 주로 **텍스트 모양으로 만드는 것**

### Context의 길이 (p.43 ~ p.45)

- 과거 초기 LLM(GPT-3.5, Llama 1·2 등): 입력 길이(Context Length)가 **매우 짧음**
  - 예) GPT-3.5 Turbo (2023년 2월) → **16K 토큰 한계**
- 모델마다 Context 길이 지원 범위가 다름
- 긴 Context를 처리하는 모델의 **실제 능력에는 한계가 있음**
- → **전체 문서를 입력하는 것은 효과적이지 못함**

### 노동법 관련 질문 예시 (p.47)

- 질문 분류: "노동법 질문" → 전체 법전 Context를 넣어 답변 요청 → **효과적이지 못함**
- → 필요한 부분만 검색 → **청킹(Chunking)**

### 청킹(Chunking) (p.49)

- 예) 매일 기록되는 일지
- **단위화 가능**
- **LLM이 내용을 딱 보고 이해 가능**

### 일반적인 텍스트의 경우 (p.51)

- **문자나 토큰 단위의 청킹 수행**

---

## 2.2 Indexing: 적절한 청크 사이즈 선택하기 (p.53 ~ p.59)

### 청킹 하기 (p.53)

예시 텍스트 ("RAG의 이해"):
- 1번 청크: "RAG를 활용하면, LLM이 사용자의 입력에 대해 보다 정확한 답변을 생성하도록 유도할 수 있습니다."
- 2번 청크: "그 이유는 프롬프트 내에 데이터베이스의 내용을 포함하여 이를 활용할 수 있기 때문입니다."
- 3번 청크: "데이터베이스는 청크 단위로 저장되며, 적절한 청크 사이즈를 찾는 것이 중요합니다."
- 예) "RAG 활용 시 LLM 답변이 정확해지는 이유는 무엇인가?" → 청크 크기가 작을 경우 **주변 정보 활용 어려움**

### 영화 데이터 임베딩 예시 (p.55)

| 방식 | 결과 |
|:---:|:---:|
| 결론만 임베딩 | **정확도 높음** |
| 결론 + 참고문헌 전체 임베딩 | **정확도 및 검색 성능 감소** |

### 청크의 크기가 클 경우 (p.57)

- 정확성이 감소하여 **검색 성능 저하**
- LLM이 필요 없는 정보까지 보게 되어 **성능 저하**

### 청크 오버랩 (p.59)

- 청크 번호가 있을 때 → 연결 시 **의미 이해 가능**
- 큰 청크 + 오버랩 → **의미 보존**
- 오버랩 비율: 일반적으로 전체 크기의 **10~20%**
- → 경우에 따라 달라지는 부분 존재

### HyDE

- **대중적으로 많이 쓰이는 방식**
- 실제 논문에서도 **성능을 올릴 수 있다고 평가**
- GPT가 전혀 모르는 얘기일 때 HyDE 불가능 → **Step back 필요**
- **프로세싱의 모든 단계에 LLM 사용 가능**

## 7.5 Searching / Augmenting: 찾아서 프롬프트 만들기 (p.125)

### Hybrid Search (종합 평균)

- **BM25 Top K** (키워드 기반)
- **Cosine Similarity Top K** (의미 기반)

## 7.6 Context Optimization: 정보 가공하고 배열하기 (p.130)

### Top K와 Context 길이의 관계

- **Top K에 정답이 포함되어 있어도 전체 Context 길이가 증가하면 성능이 감소하는 현상**

| 임베딩 리트리버 | 키워드 리트리버 |
|:---:|:---:|
| 검색을 많이 하면 **유사한 오답 생성** | 뒤로 갈수록 **오류 다수 생성** |

- 비교 모델: Gemma-7B-Chat, Gemma-2-9B-Chat, Mistral-Nemo-12B-Instruct, Gemini-1.5-Pro

## 7.7 Reordering: 중요 정보를 어디에? (p.135)

### Reordering

- 리랭커를 가지고 **점수를 얻으면 점수 기반 순서 변경**
- 중요 정보를 **맨 앞/맨 뒤에 지그재그로 배치**하거나, 관련 없는 정보를 필터링
- 특정 문제에 대해서는 **원래 순서대로의 배치가 유용**할 수 있음

# 📋 핵심 요약

## RAG의 핵심 개념

| 항목 | 설명 |
|------|------|
| **RAG 정의** | Retrieval(검색) + Augmented(증강) + Generation(생성) |
| **목적** | LLM의 Hallucination 완화, 최신/내부 데이터 활용 |
| **핵심** | 정확한 Context 입력 |
| **프롬프트 구조** | Instruction + Context + Question |

## RAG 5-step Process

| 단계 | 내용 | 핵심 포인트 |
|:---:|------|------|
| **1. Indexing** | DB 구성, 데이터 준비 | 청킹, 오버랩(10~20%) |
| **2. Processing** | 질의 전처리 | 맥락화, 쿼리 보완 |
| **3. Searching** | 적합한 데이터 검색 | Semantic + Lexical 검색 |
| **4. Augmenting** | RAG 프롬프트 생성 | Hybrid Search(BM25 + Cosine) |
| **5. Generating** | LLM 답변 생성 | Context Optimization |

## RAG vs 파인 튜닝 비교

| 구분 | RAG | 파인 튜닝 |
|------|------|------|
| **방식** | In-Context Learning | Learning (모델 파라미터 변경) |
| **지식 저장** | 외부 DB | 모델 내부 |
| **비용** | 벡터 DB 구축 | GPU + 학습 시간 |
| **데이터 변경** | DB만 수정하면 됨 | Machine-unlearning 어려움 |
| **환각** | 실제 문서 기반으로 감소 | 학습 잘 되어도 발생 가능 |
| **단점** | Search/Context 처리 시간 | 확장 어려움, 비용 큼 |

## 성능 고도화 기법

| 기법 | 설명 |
|------|------|
| **Small2Big Chunking** | 작은 청크로 검색, 큰 청크로 컨텍스트 제공 |
| **Sliding Window** | 위아래 내용 확장 |
| **Contextual Retrieval** | 각 청크에 문맥 정보 추가하여 독자적 의미 부여 |
| **HyDE** | 가상 문서 임베딩으로 검색 성능 향상 |
| **Hybrid Search** | BM25 + Cosine Similarity 종합 |
| **Reordering** | 리랭커 점수 기반 순서 변경, 중요 정보 앞/뒤 배치 |
| **Context Optimization** | Top K 결과의 정보 가공 및 배열 최적화 |
