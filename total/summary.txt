# 1장. 프롬프트 엔지니어링 기초 (p.1~p.47)

## 1.1 프롬프트 엔지니어링 기본 정의 (p.2~p.3)

- **정의**: 언어 모델에서 원하는 결과를 얻기 위한 프롬프트 작성 기술
- 주어진 Input에 대해 **높은 확률의 다음 토큰**을 출력하는 언어 모델의 **Context 생성 기술**
- LLM의 작동 원리를 고려하여 **좋은 프롬프트의 특징을 이해·활용**하는 기술
- **핵심 개념**: 같은 질문이라도 Input A → 결과 A, Input B → 결과 B로 다른 결과를 생성 (프롬프트에 따라 출력이 달라짐)

## 1.2 프롬프트 엔지니어링의 변천사 (2023~2025?) (p.4~p.5)

### ChatGPT 초창기 (2023)
- 프롬프트 엔지니어가 연봉 33만5천 달러(약 4억4천만원)로 주목받는 직업
- SBS Biz 기사: "AI 덕분에 뜬 연봉 4억4천만원인 '이 직업'...나도 해볼까?"

### 2024
- AI타임스: "몇 년 뒤 프롬프트 엔지니어링은 사라질 것"
- "2026년이나 2027년에 프롬프트 엔지니어라는 직업이 남아 있을까"

### 2025
- WSJ: "2023년 가장 인기 있는 AI 직업은 이미 구식이다"
- 프롬프트 엔지니어링이 가장 인기 있는 직업 중 하나가 될 것으로 예상되었으나 변화 중

## 1.3 프롬프트 엔지니어링은 2.5년 만에 어떻게 바뀌었는가? (p.6~p.9)

### 모델의 고도화로 인한 프롬프트 엔지니어링 기술의 변화

#### ❶ 프롬프트 단순화 (p.7~p.8)
- **예전 모델**: 정보의 단순 나열 → 원하는 포맷에 대한 구체적 요청 필요
- **최근 모델**: 정보 요약 및 구조화 → 정보의 사용 목적에 따라 적합한 포맷으로 출력
- 최근의 고성능 LLM 모델들은 프롬프트의 품질에 크게 영향 받지 않음
- 스스로 **Reasoning**하는 과정을 통해 좋은 답변을 생성

#### ❷ 메타 프롬프팅(Meta Prompting) 성능 향상 (p.8)
- LLM 자체가 다양한 프롬프트 작성 방식에 대해 학습되어 있음

#### 직무에서 역량으로 (p.9)
- 특정 직군에 맡기기보다, **내부 교육을 통해 모든 구성원의 역량**을 기르는 쪽으로 변화
- **하지만 고급 프롬프트 구성은 여전히 어려운 영역**
- 프롬프트 품질에 따른 성능 범위 그래프:
  - 예전 모델: 넓은 성능 범위 (프롬프트 품질 영향 큼)
  - 최근 모델: LLM의 메타 프롬프팅이 기본 커버 + 엔지니어의 경험과 노하우로 추가 성능

## 1.4 P.E의 확장: Context Engineering (p.10~p.12)

- 출력 구조화, 검색결과 연계(RAG), 메모리 활용 등 **맥락 설계 필요**
- **Context의 구성 요소** (다이어그램):
  - **Instructions / System Prompt** — LLM에 지시사항, 정보 제공
  - **User Prompt** — LLM에 지시사항, 정보 제공
  - **Long-Term Memory** — 어떻게 저장하는지가 중요
  - **State / History (Short-Term Memory)** — 어떻게 저장하는지가 중요
  - **Retrieved Information (RAG)** — 어떻게 저장하는지가 중요
  - **Available Tools** — 어떻게 저장하는지가 중요
  - **Structured Output** — 어떻게 저장하는지가 중요

## 1.5 Google Gemini Prompting Guide 101 (p.13~p.16)

### 효과적인 Prompt를 위한 4가지 요소 → Prompt Engineering Frameworks

| **Task** | 명확한 작업내용 제시 | — |

  - 역할을 부여하지 않아도 결과가 잘 나옴

## 1.6 Better Zero-Shot Reasoning with Role-Play Prompting (p.17~p.20)

### 핵심 Idea
- 역할 부여 프롬프팅은 추론 성능을 효과적으로 향상시킴
- 과거 명칭: **Role-Play**

### 다양한 카테고리의 Task 실험 결과 (p.18)
- "당신은 우수한 수학 선생님입니다." → 산술
- "당신은 퀴즈쇼의 참가자입니다!" → 상식
- "당신은 동전입니다. 사람들이 당신을 뒤집거나 굴립니다!" → Symbolic 추론
- "당신은 날짜 계산을 가르치는 선생님입니다." → 기타

### 실험 결과 (p.19)
- **GPT-3.5-Turbo + Llama 2-Chat(7,13,70B)** 에서 Role-Play Prompting이 **Zero-Shot-CoT보다 우수**

### 논문 해석: 왜 성능이 향상되었을까? (p.20)
- 전문가 역할에 맞는 방식의 설명
- 전문가 역할이 지식의 인출 촉진
- 일관적인 추론 패턴 생성

| Method | MultiArith | GSM8K | AddSub | AQUA | SingleEq | SVAMP |
|---|---|---|---|---|---|---|
| Few-Shot-CoT | 97.7 | 76.9 | 93.9 | 59.4 | 98.8 | 82.2 |
| Zero-Shot | 97.3 | 76 | 88.6 | 53.5 | 98.2 | 75.3 |
| Zero-Shot-CoT | 95 | 79.6 | 86.6 | 53.9 | 96.9 | 76.3 |
| **Role-Play Prompting CoT in Zero-Shot** | **97** | **78.2** | **91.1** | **63.8** | **98** | **83.8** |

## 1.7 Personas in System Prompts Do Not Improve Performances of LLMs (p.21~p.24)

### 핵심 Idea
- 시스템 프롬프트의 페르소나가 **성능 향상에 도움되지 않음**
- 논문: EMNLP 2024 Finding — 다양한 유형의 관계와 전문 분야를 고려한 162개의 역할 설정

### MMLU (Massive Multitask Language Understanding) 결과 (p.22)
- 대부분의 페르소나가 유의미하게 긍정적인 영향을 주지 않았으며, 일부 경우에는 부정적
- 'In-Domain' 페르소나를 정확히 넣는 경우에는 아주 미미한 성능 향상 확인
- 실험: "You are a [mother / lawyer / chatbot]" + 질문 → LLM → ?

### 두 논문의 비교 (p.23)
- **평가 방법의 차이**: NAACL 2024 논문은 논리적 추론이 필요한 문제 위주 실험(풀이 과정 중요), EMNLP 2024 Finding 논문은 지식 평가 위주의 실험(지식 보유 여부 중요)
- **평가 모델의 차이**: NAACL 2024는 GPT-3.5-Turbo, Llama 2-Chat / EMNLP 2024 Finding은 Qwen 2.5, Llama 3, Flan-T5-11B

### Persona 연구의 시사점 (p.24)
- **Thought 1**: 모델의 학습 방식 영향 — 최신의 모델은 페르소나 없이도 지식 베이스를 잘 이끌어 냄
- **Thought 2**: 답변의 형식 차원에서 여전히 유효 — 주어진 사람의 형식에 맞춰 답변해야 함

## 1.9 대표적 Prompt Engineering Frameworks (p.26~p.27)

### 이름은 다르지만 역할은 대부분 동일함
- **RACE Framework**: Role, Action, Context, Execute
- **TRACE Framework**: Task, Requirements, Audience, Context, Evaluation
- **ROSES Framework**: Role, Objective, Style, Example, Scenario
- **APE Framework**: Action, Purpose, Expectation

> **핵심**: 역할, 정보, 형식을 제대로 입력하는 것이 중요

### 예시: 4개의 요소를 포함한 프롬프트 (최소 75단어) (p.27)
- 당신은 기업의 COO입니다. 분기 미팅을 주최하고 있습니다.
- 직원들의 어려운 질문에 대응할 수 있는 전략을 브레인스토밍하고, 답변을 작성하세요.
- 회사의 주요 이슈: 연말 투자자 리포트, CTO 퇴사, 초기 계획 대비 변화
- 각 주제에 대해 질문 2~3개씩, 자신감 있고 단호한 어조로 답변 작성
- **토큰이 인접하여 모델이 같은 맥락으로 묶어 해석할 가능성이 있음**

## 1.10 Prompt Chaining: 복잡한 작업은 나눠서 처리하기 (p.28~p.31)

### LLM은 이전 맥락을 기억하므로, 질문 쪼개기
- **단계별 대화 흐름**:
  1. Prompt: "다음 프롬프트에 회사의 주요 이슈를 전달하겠습니다. 준비됐으면 '예'라고 대답하세요."
  2. LLM: "예. 회사의 주요 이슈들을 저에게 전달하면, 주제별로 Q&A 형식을 작성해 드리겠습니다."
  3. Prompt: (연말 투자자 리포트 + 자세한 내용 첨부)
  4. LLM: (연말 투자자 리포트에 대한 답변)

### Prompt Chaining의 장점 (p.31)
- 불필요한 정보 제거로 **Context 길이 단축**
- **사용 효율성 향상**
- **토큰 소모 절감**
- 효율적 프롬프팅을 위해 **Context 엔지니어링도 중요**

## 1.11 프롬프트 구조화하기 (p.32~p.34)

### Prompt의 구조화 — "말하듯이 해라"

### LLM이 프롬프트에서 집중해야 할 부분 강조 (p.33)
- `#`, `---`, `[]`, `<>` 등을 이용하여 각 요소를 명확하게 분리/명시
- 예시 구조:
  - 1번: 아래의 [Context]를 활용하여 [Question]에 답하세요.
  - `---`
  - 2번: [Context]:
  - `---`
  - 3번: [Question]:
- **주의**: "나는 사람이야." / "OO을 입력하세요." 같은 내용이 지시사항에 있을 경우 혼란을 줌

### 마크다운 문법을 통한 강조 (#, **) (p.34)
- GPT를 통해 생성한 프롬프트 예시:
  - `# Steps`: 1. VOC를 분석합니다. 2. VOC 내용이 어떤 팀에 부합하는지 확인합니다. 3. 적합한 팀을 출력합니다.
  - `# Output Format`: 담당 팀: '[고객 서비스/제품 개발/운영 지원팀]'
  - `# Examples` / `# Notes`

## 1.12 Few-Shot Prompting (p.35~p.42)

### 프롬프트 내에 참고할 답변 예시(Shot) 포함 (p.35)
- GPT-3에서 주로 사용하던 방법
- 예: 호랑이: Tiger, 사자: Lion, 쌍둥이: Twin
  - 입력값에 대한 LLM의 판단 과정 필요
  - 별도 설명 없이 **예시만으로 따라하기 가능**

### GPT-3.5 삼행시 만들기 예시 (p.36~p.40)
- ❌ 규칙을 말로 설명만 할 경우 → 틀린 결과값 도출
- ✅ **예시(Shot)를 포함**할 경우:
  - 규칙 설명 + 예시(청바지: 청='청춘은', 바='바로', 지='지금') + 제시어: '지피티'
  - → 결과: 지='지금은', 피='피곤하지만', 티='티비를 켜고 쉬자' (규칙에 맞는 결과 출력)
- **토크나이저 변화**: GPT-3.5/GPT-4는 '지피티'를 7토큰으로 분리 → GPT-4o/GPT-4o mini는 한국어 음절 구분 (지, 피, 티 3토큰)
- 삼행시의 규칙은 잘 지켜졌지만...

### Few-Shot Prompting에서 주의해야 할 편향 (p.42)
- 입력된 예시에 몰입된 결과로 **편향이 생길 수 있음**
- 예시의 길이, 톤, 어휘, 주제 등 특정 방향에 과도하게 영향을 받을 수 있음
- 예: LLM에게 상담을 요청하기 위해 친절한 상담원의 대사를 예시로 입력할 경우, 예시 속 말투 외에도 상담원의 설명방식, 어휘 등 다른 부분까지 함께 적용될 수 있음

# 3장. LCEL과 구조화된 출력 (p.48~p.58)

## 3.1 LCEL: LangChain의 문법 (p.48~p.54)

### LangChain
- **Chain**이라는 구조로 LLM의 구성 요소들을 연결하고 실행
- → **LangChain 표현 언어** (LCEL: LangChain Expression Language)

### LangChain의 구조 (p.50)
- **Prompt Template** → Formatting → **LLM**
- 함수(Chain)화 후 실행

### LCEL의 핵심 개념
- `|` (파이프) 연산자를 통해 구성 요소를 연결
- **Runnable** 인터페이스: `invoke()`, `stream()`, `batch()` 등

## 3.2 LCEL 2단계: 프롬프트 | LLM | Parser (p.55~p.58)

### Parser: LLM 뒤에 연결하여, 출력 형식을 변환
- LLM을 이용해 **구조화된 출력** 생성
- → Agent, Tool과 함께 다룰 내용

### 구조화된 출력
- **StrOutputParser**: 문자열 출력
- **JsonOutputParser**: JSON 형식 출력
- **PydanticOutputParser**: Pydantic 모델 기반 구조화

# 5장. 프롬프트 엔지니어링 심화 (p.59~p.89)

## 5.1 Chain-of-Thought (CoT) (p.59~p.69)

### ❶ Standard Prompting (p.60)
- 기본적인 질문-답변 방식
- Q: "Roger has 5 tennis balls. He buys 2 more cans. Each can has 3 tennis balls. How many?"
- A: "The answer is 11."
- 이어지는 질문에 대해 → **잘못된 답변 (27)**

### ❷ Chain-of-Thought Prompting (p.65) → **과정 중시**
- 풀이 과정을 예시에 포함:
  - "Roger started with 5 balls. 2 cans of 3 tennis balls each is 6 tennis balls. 5 + 6 = 11."
- 이어지는 질문에 대해 → **정확한 답변**: "The cafeteria had 23 apples originally. They used 20. So they had 23-20=3. They bought 6 more. 3+6=9. **The answer is 9.** ✅"

## 5.2 Zero-shot Chain-of-Thought (CoT) (p.70~p.72)

### 언어 모델의 추론을 금지시킨다면?
- ChatGPT 4o 예시: 복잡한 계산 문제에 "풀이 과정을 쓰지 말고, 정답만 출력해줘."
- → **잘못된 답변**: 정답 1,002,000원 (오류)

### "단계별로 생각해봐" (Let's think step by step)
- 추론 과정을 허용하면 정확도가 크게 향상

## 5.3 Step-Back Prompting (p.73~p.76)

### 핵심 개념
- 즉시 작업에 돌입하지 말고, **먼저 전체 구조를 파악**한 후 진행

### 사용법 예시 (보고서 작성 기준) (p.75)
- ❌: "보고서 써줘"
- ✅: "보고서를 쓸 건데, 전체 구획과 구조를 알려줘"

### 사용 절차
1. LLM이 보고서의 구획과 구조를 제안
2. 사용자 구조 확인 → 각 구획에 맞춰 내용 작성

## 5.5 프롬프트 작성 Tip (p.81~p.89)

### 전문 지식보다 한국어 성능이 더 중요 (p.85)

# 8장. LangChain Tools와 Function Calling (p.97~p.126)

## 8.1 LLM의 한계 극복 3요소 (p.98~p.99)

### LLM의 외연을 확장하는 세 가지 방법
- **Retrieval**: 주어진 질문에 관해 검색 후 검색 결과에 따라 답변
- **Tools**: 외부 도구 활용 (계산기, API 호출 등)
- **Memory**: 대화 이력 및 장기 기억 관리

### 아키텍처 (다이어그램)
- IN → LLM → OUT
- LLM ↔ Retrieval / Tools / Memory (Query/Results)

## 8.2 Retrieval Augmented Generation (RAG) (p.100~p.105)

### 기존의 LLM Workflow에 검색 Tool을 추가
- LLM이 모르는 내용이나 정확하게 알지 못하는 내용에도 답변 가능
- **파이프라인**: 입력 변수 → 검색 → LLM → 결과
- 검색 ↔ 결과 분석 (피드백 루프)

## 8.3 LLM Tool Calling (p.106~p.115)

### Tool이 탑재된 LLM의 판단 (p.110)
- User: "61번 서버 상태 알려줄래?"
- **Function Call**: {Tool: Retrieve_Status, Query: '61'}
- **Result**: {"Result": 504}
- LLM: "504 Error로, 게이트웨이 타임아웃으로 추정됩니다."

### LangChain에서의 Tool 사용
- `@tool` 데코레이터를 이용한 Tool 정의
- Tool의 description이 LLM의 Tool 선택에 중요한 역할
- `bind_tools()`를 통해 LLM에 Tool 바인딩

## 8.4 대표적 Tool Calling Data (p.120~p.126)

### Glaive-function-calling-v2
- Tool Calling을 **Fine Tuning**으로 만들어줄 수 있는 Data Set
- 구조: SYSTEM → 함수 Schema → USER 질문 → ASSISTANT의 function call → FUNCTION RESPONSE → ASSISTANT 최종 답변
- LLM → function call token 출력 → Tool Call 변환

- **Thought** → **Action** → **Observation** → 반복
- LLM이 스스로 추론하며 도구를 선택하고 실행

## 9.2 다단 Tool Calling 예시 (p.136~p.145)

### 계산기와 인터넷 검색 (p.140)
- Prompt 활용으로 개선
- 예시: "티모시 살라메는 1995년생이다. 이제 1, 9, 9, 5를 차례대로 곱하자."
  - LLM → FUNCTION CALL (곱하기: 1, 9) → Tool: "9"
  - LLM → FUNCTION CALL (곱하기: 9, 9) → Tool: "81"
  - LLM → FUNCTION CALL (곱하기: 81, 5) → Tool: "405"
  - LLM: "최종답변: 티모시 살라메는 (중략) 405입니다!"

### 흐름 요약 및 계획 수립
- LLM이 단계별로 계획을 세우고 실행

## 📌 핵심 요약

### 프롬프트 엔지니어링의 핵심 기법들

| 기법 | 설명 | 특징 |
|---|---|---|
| **Zero-Shot** | 예시 없이 직접 질문 | 기본적인 방법 |
| **Few-Shot** | 예시를 포함하여 질문 | 패턴 학습 유도 |
| **Chain-of-Thought (CoT)** | 풀이 과정을 포함 | 추론 정확도 향상 |
| **Zero-shot CoT** | "단계별로 생각해봐" 추가 | 간단하지만 효과적 |
| **Role-Play Prompting** | 역할 부여 | 추론 성능 향상 (단, 최신 모델에선 효과 감소) |
| **Step-Back Prompting** | 전체 구조 먼저 파악 | 복잡한 작업에 유효 |
| **Prompt Chaining** | 작업을 나누어 순차 처리 | 토큰 절감, 정확도 향상 |

### Context Engineering의 핵심 구성 요소
1. **Instructions / System Prompt** — 지시사항
2. **User Prompt** — 사용자 입력
3. **Long-Term Memory** — 장기 기억
4. **Short-Term Memory (State/History)** — 대화 이력
5. **Retrieved Information (RAG)** — 검색 결과
6. **Available Tools** — 사용 가능 도구
7. **Structured Output** — 구조화된 출력

### LangChain 핵심 개념
- **LCEL** (LangChain Expression Language): `|` 파이프로 구성 요소 연결
- **Prompt Template → LLM → Parser** 3단계 체인
- **Tools & Function Calling**: LLM의 외부 도구 활용
- **Agents (ReAct)**: Reasoning + Acting으로 복잡한 작업 자율 수행
- **Multi Agent**: 여러 Agent가 협력하여 대규모 작업 수행
