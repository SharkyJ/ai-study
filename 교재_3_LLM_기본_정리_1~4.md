## 1.1 LLM 기반 기술 이해하기 (p.2~3)

### LLM의 특징
- 언어 패턴을 학습해 언어를 **생성**하는 모델
- 현재 LLM의 주된 역할 = **언어 생성**
- LLM을 구성하는 핵심 모델 = **Transformer 구조**

### 핵심 구조 및 진행 방향
- Transformer 구조와 원리 소개
- 파생 모델인 GPT의 시작과 최신 LLM 발전 과정까지 탐구

---

## 1.2 기계 학습 - Concept (p.4~5)

### 데이터를 이용해서 데이터의 규칙을 학습하기

**분류(Classification) 방식 예시:**
- 입력: 개·고양이 사진 → 픽셀 값
- 과정: 모델이 연산(곱·합·활성화 함수) 거침
- 출력: 결과가 1 → 고양이, 0 → 개

**예측(Regression) 방식 예시:**
- 입력: 일정 기간(예: 한 달)의 주가 정보·공시
- 출력: 오늘의 종가 예측
- 학습 목적 = 과거 데이터 기반으로 내일의 값 예측

---

## 1.3 데이터의 패턴 (p.6~10)

### 패턴의 정의 (p.6)
- **데이터 분포 = 패턴**
- 데이터가 형성하는 **통계적 분포**

### 선형 회귀 예시 (p.7)
- 빨간 점(데이터), 파란 선(1차 회귀선)
- 회귀선 = 데이터 분포 이해 모델
- 파란 선이 빨간 점들의 분포를 설명
- **오차 최소화**: 최소자승법, R² 등 활용

### 예측과 생성의 차이 (p.8)
- **예측**: 기존 데이터 분포 이해 → 새로운 입력(X)에 대한 출력(Y) 추론
- **생성**: 데이터가 없어도 모델만으로 새로운 점(X=1 → Y=3 등) 생성
- **생성 AI의 원리** = 패턴 이해 + 새로운 데이터 생성

### 언어 모델(LLM)의 적용 (p.9~10)
- 언어 = 훨씬 더 복잡한 데이터 구조
- 단순 선형 모델(a, b 파라미터)와 달리 LLM 파라미터 = **수억~수천억 개**
- 언어 데이터의 다양성과 복잡성·유창성 반영 필요
- **학습 데이터 규모**: Qwen 3 → 약 37TB 텍스트(실제 용량~100TB)
- 방대한 데이터 속 언어 패턴 학습 → 인간처럼 자연스러운 언어 생성

## 1.4 자연어 학습의 어려움 (p.11~13)

### 자연어 vs 수치형 데이터 (p.11)
- 자연어 데이터 ≠ 픽셀 값·수치형 데이터
- 기계학습 모델을 만들어도 **+, -가 가능한 입력으로 만드는 과정 선행 필요**

### 구체적인 어려움 (p.12)
- 숫자 연산이 불가능하므로 → **숫자(벡터)로 변환 필요**
- 픽셀 데이터처럼 **길이가 일정하지 않음**

### 어순의 중요성 (p.13)
- "A는 B를 좋아한다" ≠ "B는 A를 좋아한다"
- **자연어 학습 = 모델 설계 전 사전 세팅 과정 필수**

---

## 1.5 임베딩: 추상적 데이터 컴퓨터에게 이해시키기 (p.14~20)

### 임베딩(Embedding)과 Word2Vec의 개념 (p.14~16)
- 자연어 의미를 컴퓨터가 이해할 수 있도록 **벡터로 변환하는 과정**
- 언어뿐 아니라 그래프, 단어, 그림, 사운드 등 추상적 데이터도 벡터로 변환 가능

**Word Embedding (p.15):**
- 단어를 **수백~수천 차원의 벡터**로 변환
- 유사 단어 벡터 공간에서 가깝게 배치 = **Word2Vec**

**Word2Vec (p.16):**
- 구글에서 **2013년** 개발
- 예: "나는 커피를 좋아한다, 나는 우유를 좋아한다, 나는 차를 좋아한다..."
  - "커피를", "우유를", "홍차를", "녹차를" → **비슷한 벡터 값**

### Word2Vec의 연산적 특성 (p.17~20)
- **성별 관계**: King → Queen, Man → Woman (같은 방향 벡터)
- **시제 관계**: Draw → Drew, Go → Went (같은 방향 벡터)
- Woman - Man = Queen - King = 같은 Vector (성별 차이 벡터)
- **수도-국가 연산**: 서울 - 코리아 = X, 로마 - 이탈리아 = X, 독일 + X = **베를린**
- 자연어처리 초반에 **가장 널리 사용된 방식**

---

## 1.6 Word Embedding의 한계점 (p.21)

### Word2Vec의 활용과 한계
- 딥 러닝 자연어 처리 초기에 많이 활용했으나, 한계가 존재

| 한계 | 설명 |
|---|---|
| **동음이의어 처리 불가** | 하나의 벡터에 하나의 단어를 할당 → 다의어 처리 어려움. 여러 의미를 무게중심으로 처리 → 의미 정확도 떨어짐 |
| **오타 처리 불가** | King → KNG 같은 오타 발생 시 인식 불가. UNK(unknown) 토큰으로 처리 → 부분적 이해 불가능 |
| **확장성 문제** | 단어 수가 많아질수록 모든 단어 벡터화 불가능. 공간 부족, 확장성에 문제 발생 |

---

## 1.7 맥락(Context) (p.22)

### 오타 처리와 의미 이해
- "I go to school" → "나는 학교에 간다"라는 뜻
- school이 schol로 오타남 → schol = school일 것이라 유추
  - **철자 유사성**: S, OL 조합 보고 알 수 있음
  - **주변 맥락**: I, to, go 등이 함께 있어 의미 보완
- **세부 구성 분석 및 주변 맥락(Context)과 함께 고려 필요**

---

## 1.8 토큰(Token)의 정의와 특징 (p.23~24)

### 토큰(Token)의 정의 (p.23)
- 단어나 텍스트 묶음을 더 잘게 나눈 단위
- 단어보다 작고, 글자 하나보다는 큼
- 예: "고" = g + o → 하지만 "우어먼(woman)"의 o와는 다름
- "Tokenization" = "Token" + "ization" → 조합으로 이해
- "Tokenizer, Tokenized" 등의 표현도 **2개 토큰으로 처리 가능**

### 토큰 기반 임베딩 (p.24)
- 임베딩을 **단어가 아닌 토큰 레벨**로 계산
- 토큰 간 관계를 분석해 **맥락(Context)을 이해**
- → **트랜스포머(Transformer)**

---

## 1.9 트랜스포머(Transformer) 소개 (p.25~28)

### 트랜스포머의 등장 (p.25)
- **2017년** 구글에서 개발
- 인용수 약 **19만 번** → 기념비적 성과
- 10년 가까이 지난 지금도 대부분 LLM 구조가 트랜스포머 기반
- 논문: **"Attention is All You Need"**

### 당시 과제와 한계 극복 (p.26)
- 2017년은 LLM 시대 이전
- 주요 과제 = **문장 변환(Sequence-to-Sequence)**
  - 예: "나는 집에 간다" → "I go home"
- 기존 모델(RNN 등) 한계: 문장이 길면 내용 잊음
- Transformer는 **긴 문장도 내용 보존 가능** → 긴 Context 처리 강점

### 초기의 역할 (p.27)
- 최초 트랜스포머의 목적은 **Sequence to Sequence**
- **인코더(Encoder)**: 입력 문장을 받아 의미를 벡터로 계산
- **디코더(Decoder)**: 인코더 결과를 받아 출력 문장 생성
- 출력 생성은 **순차적(토큰 단위 하나씩 생성)**

### 내부 모듈과 확장성 (p.28)
- Attention, Feed Forward, Linear 등 모듈들을 겹겹이 쌓아올린 것이 Transformer
- 이후 확장·복잡화 → **현대 LLM의 기반**이 됨
- 인코더 구조 → **BERT** 모델로 발전
- 디코더 구조 → **GPT** 같은 LLM으로 발전

---

## 1.10 토큰화: 텍스트를 처리하는 가장 작은 단위 (p.29~32)

### 토큰화의 기본 개념과 사례 (p.29~30)
- 텍스트를 작은 단위로 쪼개는 과정 = **토큰화(Tokenization)**
- 예: Generative Pre-trained Transformer (GPT) → 토큰1~5로 분리
- 단어 그대로 저장하는 것보다 공간 효율 ↑, 다양한 처리 가능

**GPT 3.5 / GPT 4 기준 (p.30):**
- 한국어 "지피티" → 총 7개 토큰으로 분리
- 이유: '피', '티' 같은 단어가 토큰 목록에 없어서 바이트 단위로 쪼갬
- 문제: 실제 글자는 3개인데 LLM은 7개로 인식 → **비효율 발생**

**GPT-4o & GPT-4o mini 기준 (p.31):**
- "지피티" → 개선된 토큰화로 효율 향상
- 토큰 개수가 많아지면 → Context 길이 제한이 빨리 소진, 처리 속도 저하

### 토큰화의 언어별 차이 (p.32)
- **영어**: 토큰 압축률 높음 → 보통 4:1 ~ 5:1
- **한국어**: 압축률 낮음 → 2:1 ~ 1.8:1 수준
- → **프롬프트는 영어로 쓰는 게 유리함** (영어 토큰화가 효율적이라 더 많은 맥락 처리 가능)

# 📖 Chapter 2. Transformer와 LLM의 탄생 (p.33~72)

---

## 2.1 Transformer: 입/출력 비교 (p.34~38)

### Transformer 구조 (p.34)
- 문장 → 토큰화 → 토큰별 임베딩
- 문장을 토큰으로 쪼개고 토큰별 벡터 계산

### Encoder 입/출력 (p.35)
- 토큰별 임베딩 벡터("나는", "밥을", "먹었다") → **Encoder** → 토큰별 맥락 벡터(병렬적 생성) → **Decoder**

### RNN vs Transformer (p.36)
- **RNN**: 나는 → 밥을 → 먹었다 → 순차적 처리. 문장이 길어지면 **앞의 내용 손실**
- **Transformer**: 각 토큰들을 **병렬적으로 개수를 유지하면서 생성**

### Decoder 입/출력 (p.37~38)
- Decoder 입력: Encoder의 출력값 + 현재까지의 출력 문장
- 처음에는 `<BOS>`만 입력(번역을 시작하라는 의미)
- `<BOS>` → I → `<BOS> I` → Had → `<BOS> I Had` → Meal → `<BOS> I Had Meal` → `<EOS>`
- **자기 회귀적(Auto Regressive) 출력 방식**

---

## 2.2 Positional Encoding (p.39~43)

### Encoder/Decoder 원리 (p.39~40)
- 병렬 실행이므로 Encoder는 **각각의 순서를 알 수 없음**
- 순서를 반영하지 못하기 때문에 그냥 처리하면 **토큰들의 묶음처럼** 처리하게 됨

### 위치 인코딩 벡터를 통해 토큰의 위치 구분하기 (p.41~43)
- "나는 학교에 가고, 강아지는 산책을 간다" vs "나는 산책을 가고, 강아지는 학교에 간다"
  - 두 문장의 구분이 필요 → 각 토큰에 위치 번호(+1, +2, +3, ...) 부여
  - "나는" "학교에" 갈 확률이 높음 (거리가 가까움)

- **RoPE**: 각각의 벡터의 임베딩 벡터를 조금씩 회전시키는 **회전 기반 방법** 사용
- 목적: Input을 받았을 때 각각의 위치 기억, Decoder에서 중간 출력을 받았을 때 각각의 위치 기억
- 토큰 목록을 받았을 때 각각의 위치를 전달해주기 위한 정보를 **추가로 더해줌**

---

## 2.3 Transformer Layer의 핵심: Self-Attention & FFNN (p.44~48)

### Transformer가 입력 데이터를 처리하는 방식 (p.44)
- **Self-Attention**: 맥락을 반영하여, 어디에 집중할 것인가?
- **FFNN(Feed-Forward Neural Network, MLP)**: 어떤 지식을 활용하여 이해할 것인가?
- **Nx**: Self-Attention, FFNN의 입/출력을 **N번 반복**
- **Add & Norm**: 통과한 결과와 통과하기 전 결과를 합쳐서 평균을 냄

### Linear & Softmax (p.45)
- **Linear**: 벡터를 가지고 토큰의 가짓수만큼의 벡터로 다시 확장
- **Softmax**: Linear 결과를 확률분포로 변경한 뒤, 확률분포로 바꾸면 **샘플링을 통해 단어를 선택** 후 생성

### Encoder의 FFNN (p.48)
- Self-Attention을 통과한 후 FFNN을 거쳐 최종 맥락 벡터 생성

---

## 2.4 Multi-head Attention (p.52~55)

### Encoder 구조 이해 시 고려할 점 (p.52)
- 토큰과 토큰의 연산을 계산할 때 **한 가지 맥락이 아니라 여러 맥락을 고려**해야 함

---

## 2.5 Decoder 구조 (p.56~63)

### Decoder의 역할 (p.56)
- **목적**: `<BOS> I had meal` 문장 출력
- **Input**: Encoder 결과 + 출력 문장의 앞부분
- **Output**: 다음 토큰의 확률 분포

---

## 2.6 Decoding: Masking (p.60~63)

### Encoder Self-Attention (p.60)
- 앞뒤 단어끼리 연산
- **N² 만큼 연산**

### Decoder의 Masked Self-Attention
- Decoder에서는 미래 토큰을 볼 수 없도록 **마스킹** 처리

---

## 2.7 Transformer 성공의 영향 (p.64~72)

### Transformer 구조 (End to End 구조) (p.64)
- Input → Masked Self-Attention → Encoder-Decoder Attention → FFNN → Output

### GPT 구조 (Decoder 구조만 활용) (p.68)
- Encoder 부분과 Encoder-Decoder Attention을 **제거**
- Decoder만으로 텍스트 생성

### Transformer의 영향 (p.72)
- Transformer → **번역**
- 자연어 모델(BERT, GPT) → 다양한 NLP Task

# 📖 Chapter 3. LLM의 발전 과정 (p.73~120)

---

## 3.1 Transformer 성공의 영향 (p.74~75)

### Transformer의 구조
- **Encoder**: 토큰의 길이가 길어져도 내용을 잃어버리지 않음
- **Decoder**: 중요도를 잘 이해하고 출력 문장을 자연스럽게 생성

### BERT (2018) vs GPT (2018) (p.75)
- **BERT**: Encoder 구조를 활용하여 **문장 의미 파악** → 빈칸을 만든 다음 예측하는 문제, 문장들의 관계를 예측하여 벡터 출력
- **GPT**: Decoder 구조를 활용하여 **텍스트 생성** → 다음 단어를 예측하여 문장을 작성

---

## 3.2 GPT: Next Token Prediction (p.76~77)

### 문맥을 고려하여 주어진 문장의 다음 단어 예측하기
- **Generative Pre-trained Transformer** (2018)
- Input: `<BOS>` → 나는 → 오늘 → 너무 → 화가 → 났다 → 그 → 이유는 → ...
- Output: 나는 → 오늘 → 너무 → 화가 → 났다 → 그 → 이유는 → ... → `<EOS>`

---

## 3.3 Transformer 기반의 "사전 학습" 모델: BERT와 GPT (p.78~83)

### 리뷰 분류 문제 (p.78)
- "리뷰 분류"라는 단일 Task만을 목적으로 학습
- 딥러닝 모델 초기화 → 학습할 리뷰 준비 → 0 vs 1 학습

### BERT와 GPT (p.80)
- 특정 Task가 아닌, **언어의 패턴 자체를 학습하고자 함**
- 약간의 추가 학습만 거쳐도 단일 Task 모델처럼 뛰어난 성능을 보임

---

## 3.4 GPT부터 GPT-3까지 (p.84~91)

### GPT-1 vs GPT-2 (p.84)
- **GPT-1**: 요약하려면 요약 데이터를 가르쳐야 함
- **GPT-2**: 긴 글 뒤에 TL;DR을 붙이면 요약을 만듦

### GPT-3: 175B (p.88)
- **In-Context Learning 능력 발견**
  - 입력 데이터를 주고 다음에 관련 내용을 물어봤을 때 해당 데이터를 바탕으로 답변
  - 예: 메뉴별 가격과 함께 'A 메뉴의 가격은?'이라고 입력 시 '3만 7천원이다.'와 같이 답변

---

## 3.5 GPT-3.5부터 GPT-4까지 (p.92~113)

### GPT-3.5: 175B (p.92~105)

**SFT + RLHF (Supervised Fine Tuning + Reinforcement Learning with Human Feedback):**

**Step 1 - SFT (p.96):**
- 시범 데이터를 수집하고 지도학습 정책 훈련
- "닐 암스트롱의 달 착륙은" 이라고 입력 시 **정확하게 답변 가능**

**Step 2 - Reward Model 학습 (p.100):**
- 프롬프트에 대해 여러 모델 출력을 생성
- 라벨러가 출력들을 최고에서 최저로 순위 매김
- 예: A(중력을 설명해 주세요), B(전쟁을 설명해 주세요), C(달은 지구의 자연 위성입니다), D(사람들이 달에 가서 사진을 찍었습니다)
- 이 데이터로 Reward Model 훈련

**Step 3 - PPO 강화학습 (p.105):**
- 새로운 프롬프트에 대해 정책이 출력 생성
- Reward Model이 보상 점수 계산
- **Reward 점수가 높은 방향으로 다시 학습**

### GPT-4: 1760B (p.110)
- **MoE 구조**: Self-Attention 공유 + 여러 개의 MLP 중 **일부만 활성화**
- Router가 각 토큰에 대해 어떤 FFN을 활성화할지 결정

---

## 3.6 GPT-4o (p.115~118)

### GPT to GPT-4o: Early Fusion
- **GPT4**: Vision Encoder를 이용하여 이미지 인코딩 생성 후 결합
  - 이미지가 이해한 벡터를 언어 모델의 벡터로 변환시켜주는 **Projection layer** 필요

---

## 3.7 왜 지금은 Decoder 기반의 모델이 분류도 잘 하나요? (p.120)

- Encoder 모델은 사용 방식이 제한적
- 어느 정도의 파라미터가 주어지고 데이터를 충분히 학습하면, **Decoder 모델도 높은 수준의 이해력**을 갖게 됨
- 최근의 Decoder 기반 모델은 **다양한 언어 Task에 모두 활용 가능**

# 📌 핵심 요약

## 전체 흐름 정리

```
Word2Vec(2013) → Transformer(2017) → BERT/GPT(2018)
→ GPT-2 → GPT-3(175B, ICL 발견) → GPT-3.5(SFT+RLHF)
→ GPT-4(1760B, MoE) → GPT-4o(Early Fusion, 멀티모달)
```

## 핵심 개념 요약

| 개념 | 설명 |
|---|---|
| **임베딩** | 자연어를 벡터(숫자)로 변환하는 과정 |
| **Word2Vec** | 단어를 벡터 공간에 배치, 유사 단어는 가까이 |
| **토큰** | 단어보다 작은 텍스트 처리 단위 |
| **Transformer** | Self-Attention 기반 병렬 처리 모델 (2017) |
| **Self-Attention** | 토큰 간 관계를 계산하여 맥락 반영 |
| **FFNN** | 지식을 활용하여 이해하는 계층 |
| **Positional Encoding** | 토큰의 순서 정보를 부여 (RoPE 등) |
| **Masking** | Decoder에서 미래 토큰 참조 방지 |
| **Encoder** | 입력을 이해하는 구조 (BERT) |
| **Decoder** | 출력을 생성하는 구조 (GPT) |
| **SFT** | 시범 데이터로 지도학습 미세조정 |
| **RLHF** | 인간 피드백 기반 강화학습 |
| **MoE** | 여러 전문가 중 일부만 활성화하는 구조 |
| **ICL** | In-Context Learning, 입력 데이터 기반 답변 |
